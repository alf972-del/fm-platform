Bases de Datos & Almacenamiento
Persistencia multi-modelo
PostgreSQL 17
+ RLS
Base de datos principal
Multi-tenant con Row Level Security nativo por tenant_id. Un esquema unificado, aislamiento garantizado a nivel de BD. Prisma ORM para type-safety. JSONB para campos dinámicos por tipo de centro.
✓ Recomendado RLS multi-tenant Prisma ORM JSONB
TimescaleDB
Extensión PG
Series temporales IoT
Extensión de PostgreSQL para datos de sensores (temperatura, consumo, pH). Compresión automática, aggregation continua, queries rápidas sobre millones de puntos de datos.
✓ Recomendado Compresión nativa Mismo PG
Redis 7
Stack

Autenticación & Seguridad
Multi-tenant seguro
Keycloak
v25
Identity Provider central
SSO corporativo con SAML 2.0 y OIDC. MFA nativo, gestión de roles por tenant, federación con Active Directory de clientes. Self-hosted, sin lock-in.
✓ Recomendado SAML + OIDC MFA nativo AD federation
CASL
Node lib
Autorización granular (RBAC)
Control de acceso basado en roles y atributos. Cada acción (crear OT, ver contrato, aprobar presupuesto) con permisos configurables por rol y por tenant.
RBAC + ABAC
PostgreSQL RLS
Row Level Security
Aislamiento de datos
Cada query filtra automáticamente por tenant_id a nivel de base de datos. Imposible que un tenant acceda a datos de otro, incluso con bug en la app.
✓ CríticoCapa extra de seguridad

07
Decisiones Clave — Alternativas Evaluadas
Decisión Elegido Alternativa Razón del descarte Veredicto
Lenguaje backend TypeScript / Node Python (FastAPI) / Go Node comparte código con frontend RN; equipo full-stack más pequeño. FastAPI si el equipo es Python-heavy. Sí
Mobile framework React Native + Expo Flutter Flutter produce mejores apps nativas pero no comparte código con el web. RN permite team full-stack JS. Sí
Offline móvil WatermelonDB Realm / SQLite directo Realm requiere backend propio (Atlas). SQLite directo sin sync. WatermelonDB tiene sync incremental listo. Sí
Multi-tenancy Schema único + RLS Schema por tenant Schema por tenant escala mal operativamente con 50+ clientes (migrations, backups). RLS es más mantenible. Sí
Auth Keycloak Auth0 / Supabase Auth Auth0 caro a escala; Supabase Auth simple pero limitado en roles complejos. Keycloak self-hosted, gratuito, SAML nativo. Sí
Base de datos PostgreSQL MongoDB FM es datos muy relacionales (activos → OTs → técnicos → SLAs). MongoDB añade complejidad de joins sin beneficio real. Sí
BI inicial Metabase embed PowerBI / Tableau PowerBI/Tableau caros y requieren licencias por usuario. Metabase OSS embebido es suficiente hasta ~$500K ARR. Fase 1 OK
Infra MVP Railway PaaS AWS EKS / K8s propio K8s propio requiere DevOps dedicado desde día 1. Railway reduce complejidad operativa en fase de validación. Migrar en escala
08
Roadmap de Implementación del Stack

FASE 01 · Meses 1–3
MVP Core
  • PostgreSQL + Prisma schema base
  • NestJS API: activos, OTs, usuarios
  • React web: dashboard básico
  • RN app: OT flow técnico
  • Auth Keycloak + roles
  • Deploy Railway staging + prod

FASE 02 · Meses 4–6
Soft FM + Offline
  • WatermelonDB offline sync
  • BullMQ scheduler preventivos
  • Módulo soft FM (limpieza/NFC)
  • Socket.io tiempo real
  • Portal inquilinos (Next.js)
  • S3 / MinIO para documentos

FASE 03 · Meses 7–10
IoT + Analytics
  • TimescaleDB + MQTT broker
  • Conectores IoT sensores
  • Metabase embed reportes
  • Motor KPI automático
  • Meilisearch búsqueda
  • ESG dashboard

FASE 04 · Meses 11–14
Escala & Integraciones
  • Migración K8s / EKS
  • API pública documentada
  • Conectores ERP (SAP/Oracle)
  • BIM/IFC viewer integrado
  • IA predictiva mantenimiento
  • Marketplace integraciones
