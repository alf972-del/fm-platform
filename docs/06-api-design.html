POST /v1/work-orders/:id/transition Cambiar estado de la OT (state machine) 🔒 fm:write ▾
Transiciones válidas (State Machine)
// Estado actual → Transiciones permitidas según rol draft → [ "submit" ] // requester pending → [ "approve", "reject" ] // supervisor approved → [ "assign" ] // supervisor assigned → [ "start", "reassign" ] // technician / supervisor in_progress → [ "pause", "complete", "escalate" ] // technician completed → [ "verify", "reopen" ] // supervisor verified → [ "close" ] // supervisor / auto // cualquier estado → "cancel" (solo supervisor)
Request Body
{ "action": "start", // requerido — ver tabla arriba "comment": "Inicio diagnóstico en planta 3", // opcional "assigned_to": "u1u2u3u4-...", // requerido si action = "assign"/"reassign" "resolution": "Capacitor reemplazado", // requerido si action = "complete" "actual_cost": 320.00 // opcional si action = "complete" }
Respuestas
200 OK — Estado actualizado + webhook disparado 409 Conflict — Transición inválida desde estado actual 403 Forbidden — Rol no autorizado para esta transición

Webhooks
Notificaciones push de eventos del sistema
📡 Configuración de Webhooks
Los webhooks envían un HTTP POST firmado (HMAC-SHA256) al endpoint del cliente cuando ocurren eventos en el sistema. Incluyen reintentos exponenciales (3 intentos, 5s → 25s → 125s). Verificar firma con el header X-FM-Signature.
